# Worker coordination: heartbeats, janitor, graceful shutdown, concurrency

The worker process gains four interlocking capabilities: a 5-second heartbeat with 15-second TTL marks workers as alive; a janitor periodically scans the registry for workers whose heartbeat has expired, re-enqueuing their orphaned jobs; graceful shutdown drains in-flight work (30s timeout), then deregisters from the cluster; and a configurable concurrency model (default 5) replaces the previous single-job-at-a-time loop with a slot-based approach using a `Set<Promise>` of active jobs.

Watch for: janitor/worker race window where a slow worker might have its jobs reclaimed while still alive (confirmed); janitor does not cap how many workers or jobs it processes per tick and swallows errors from failed ticks (confirmed); concurrency default changed from 1 to 5 which is a behavioral change for existing deployments (confirmed).

## High-level view

The heartbeat uses a Redis key with TTL as a lease. The 15-second TTL against a 5-second write interval gives three missed heartbeats before expiry—tight enough to recover jobs quickly, but also tight enough that a GC-stalled worker can be declared dead while still alive.

The janitor iterates over all registered worker IDs, checks for heartbeat presence, and sequentially re-enqueues every orphaned job from dead workers. It resets job status to `pending` in both Redis and Postgres. The design has a race condition: a heartbeat TTL can expire mid-processing if a worker is alive but stalled, causing the janitor to reclaim jobs that are still being processed. There is also no error handling or concurrency guard between ticks.

Graceful shutdown waits for active promises with a 30-second hardcoded timeout (no config knob), then deregisters and deletes keys.

The concurrency model dispatches jobs into a `Set<Promise<void>>` and gates the pull loop behind a size check. When slots are full, the production loop polls every 50ms rather than using `Promise.race` on the active set (a pattern already used in the `processAvailable` test helper in the same file).

<details>
<summary>Issues (5)</summary>

1. **Heartbeat/janitor race on slow workers** — A worker under GC pressure or heavy load can miss three heartbeat writes, causing the janitor to reclaim its jobs while they are still executing. This leads to duplicate processing. Consider adding a grace period (track first-seen-missing across ticks) or a job-level claim flag.
2. **Janitor tick unbounded and overlapping** — `setInterval` fires regardless of whether the prior tick has returned. A mass worker death causes hundreds of sequential awaits in one tick, which can exceed the interval and cause concurrent overlapping ticks. Use a self-scheduling `setTimeout` pattern or gate on a running flag.
3. **Janitor swallows errors silently** — `void this.tick()` discards rejections. A partial failure (some jobs re-enqueued, then a throw) leaves the dead worker in an inconsistent state with no logging. Wrap in try/catch per-worker.
4. **Concurrency slot polling wastes cycles** — The main loop polls every 50ms when slots are full instead of using `Promise.race` on the active set. Under sustained full concurrency, this is a continuous 50ms wakeup.
5. **Shutdown timeout not configurable** — The 30-second drain timeout is hardcoded. Operators with long-running job handlers cannot tune this.

## Heartbeat-janitor race window

The janitor determines liveness with a single `EXISTS` check on the heartbeat key. If a worker's heartbeat TTL (15s) expires because the event loop was blocked for 15+ seconds (GC pause, saturated CPU, slow syscall), the janitor will treat it as dead and re-enqueue its processing list. The worker, still alive, will then complete those same jobs and attempt to mark them completed in both Postgres and Redis. The result is duplicate execution.

```
Janitor.tick():
  exists = await this.redis.exists(redisKeys.heartbeat(workerId));
  if (exists) continue;  // <-- only gate
```

Two mitigations that don't require consensus protocols: (1) Before re-enqueueing, set a "claimed-by-janitor" flag on the job and have `processOne` check it before completing; (2) Track first-seen-missing in memory and only reclaim after the heartbeat has been absent for two consecutive janitor cycles.

Confidence: **confirmed** — the race exists by construction; there is no guard against it.

## Unbounded janitor tick

```typescript
for (const workerId of workerIds) {
  // ... for each dead worker
  for (const jobId of orphanedJobs) {
    // sequential await per job
  }
}
```

If 50 workers die simultaneously (deployment rollout, node failure) each with 10 jobs, the janitor executes 500 sequential Redis+Postgres operations in one tick. `setInterval` fires regardless of whether the callback has returned, and `void this.tick()` means multiple ticks can run in parallel. Two concurrent ticks operating on the same dead worker will both read the processing list before either deletes it, causing double re-enqueue.

Confidence: **confirmed** — `setInterval` + `void this.tick()` pattern visible in `start()`.

## 50ms slot polling in the main loop

```typescript
if (this.activeJobs.size >= this.concurrency) {
  await new Promise((r) => setTimeout(r, 50));
  continue;
}
```

The `processAvailable` method in the same class demonstrates the better pattern: `await Promise.race([...this.activeJobs])` which wakes exactly when a slot frees up. At concurrency=5 with jobs averaging 100ms, the production loop wakes ~20 times per second unnecessarily.

Confidence: **confirmed** — both patterns visible in the same class.

## Janitor error handling gap

The janitor's `start()` fires ticks with `void this.tick()`, discarding the returned promise. If any Redis or Postgres call inside `tick()` throws, the error goes to the unhandled rejection handler. A partial failure leaves the dead worker partially cleaned: some jobs re-enqueued, others still in the processing list, but the worker not yet removed from the registry. The next tick will re-process this worker—but the already-recovered jobs are no longer in the processing list, so only the remaining ones get processed. This is actually safe for non-overlapping ticks, but combined with the overlapping-tick issue above, it becomes a double-enqueue vector.

Confidence: **confirmed** — `void this.tick()` discards rejections; no try/catch inside `tick()`.

## Concurrency default change from 1 to 5

The default concurrency changed from 1 to 5 (`Number(env.WORKER_CONCURRENCY ?? 1)` became `?? 5`). Existing deployments that relied on the implicit single-job semantics (sequential processing guarantees, predictable memory per worker) will see a 5x increase in parallel work without any config change. This could overwhelm downstream databases or APIs that were sized for one-at-a-time processing.

Confidence: **confirmed** — visible in the config diff.

## Test coverage gaps

Not tested: janitor behavior when a job hash is missing from Redis (orphaned ID in processing list but no corresponding job hash—`hgetall` returns `{}`); janitor behavior under concurrent ticks; worker behavior when heartbeat write fails (Redis unavailable); shutdown with the 30s timeout actually expiring (test uses a 100ms job); interaction between rate limiter and the new concurrency model (a rate-limited job pushed back to queue still occupies a slot briefly).
