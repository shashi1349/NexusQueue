# Observability dashboard: REST API, WebSocket event bus, and React UI

This change adds a real-time observability layer to NexusQueue. The server gains six new REST endpoints (queue listing, job browsing, retry, workers, DLQ inspection, bulk requeue), a WebSocket event bus backed by Redis PUB/SUB that broadcasts job lifecycle events, and CORS middleware. A separate React workspace (`dashboard/`) provides a Vite+Tailwind UI with live throughput charting, job streaming, and DLQ management.

Watch for: the Express error handler is mounted before routes, so route errors bypass it entirely (confirmed). The `cors()` call uses default config — `Access-Control-Allow-Origin: *` — which opens the mutation endpoints (`POST /jobs/:id/retry`, `POST /queues/:name/dlq/requeue`) to any origin (confirmed). The bulk requeue endpoint processes jobs sequentially with no atomicity; a partial failure leaves Redis and Postgres in inconsistent states (confirmed). The WebSocket server has no authentication or rate limiting (confirmed).

## High-level view

The REST API is a read-heavy surface built on Redis for queue/worker metadata and Postgres for job history. The design diverges per endpoint: queue stats query both stores, job listing is Postgres-only, DLQ listing is Redis-only. This asymmetry means the DLQ inspector shows stale data if Redis and Postgres drift — something the retry/requeue operations can cause when they partially fail mid-flight.

The event bus is thin: a `NexusEventBus` class wraps Redis PUB/SUB and a `ws` WebSocketServer, relaying published messages to all connected WebSocket clients with no filtering or backpressure. The producer emits `job.created` via a monkey-patched `deps.eventBus` property. Other lifecycle events (`job.started`, `job.completed`, `job.failed`, `job.dlq`) are not emitted by the worker, so the live stream shows only creation and manual retry events until the worker is wired in.

The dashboard is a standard Vite+React SPA using polling (5s intervals) for REST data and a persistent WebSocket for the live stream. The `WebSocketManager` singleton implements exponential backoff reconnection. There are no frontend tests.

The error handler ordering bug means all six new endpoints fail silently (500 from Express default behavior) rather than returning the structured `{ error: 'internal_error' }` response the code intends.

1. **Error handler mounted before routes** — Move the centralized error handler below `app.use(buildRouter(...))` in `server/src/index.ts`. Currently it never fires for route errors.
2. **CORS allows all origins on mutation endpoints** — Configure `cors()` with an explicit origin allowlist (or at minimum restrict to the dashboard's origin via `VITE_API_URL`). The retry and requeue endpoints mutate state and should not be callable from arbitrary origins.
3. **No WebSocket authentication** — Any client can connect to the WS endpoint and receive all job events. Add a token/origin check on `connection` before relaying messages.
4. **Bulk requeue is non-atomic** — A crash mid-loop in `POST /queues/:name/dlq/requeue` leaves some jobs requeued and others still in DLQ, with Redis and Postgres potentially diverged. Wrap the Postgres updates in a transaction and consider a pipeline for Redis.
5. **Incomplete event coverage** — Only `job.created` and `job.retried` events are emitted. The throughput chart depends on `job.completed` events that are never published (worker not wired to event bus).
6. **EventBus injected via monkey-patch** — `(producer as any).deps.eventBus = eventBus` bypasses the typed interface. Pass `eventBus` through the constructor or a setter to preserve type safety.
7. **No validation on `status` query parameter** — `GET /queues/:name/jobs?status=anything` accepts arbitrary strings. Invalid statuses silently return zero results. Validate against the known enum.
8. **DLQ endpoint has no pagination** — `GET /queues/:name/dlq` fetches all job IDs with `LRANGE 0 -1` and does a Redis roundtrip per job. Add limit/offset like the queue-jobs endpoint.

## Error handler ordering in Express

In `server/src/index.ts`, the error-handling middleware is registered at line 20, before `app.use(buildRouter(...))` at line 47. Express dispatches errors only to error handlers that appear *after* the middleware that threw. Any `next(err)` call in the route handlers will hit Express's built-in handler (HTML stack trace in dev, empty 500 in prod) rather than the structured JSON response the app intends.

```
app.use(express.json(...));
app.use(errorHandler);       // <-- registered here
...
app.use(buildRouter(...));   // <-- errors thrown here never reach errorHandler
```

This is a regression from the pre-Phase-5 code. Previously routes were mounted *before* the error handler, so it worked. The reordering to "mount routes after event bus is ready" introduced the bug.

## CORS and WebSocket authentication gap

`cors()` with no arguments sets `Access-Control-Allow-Origin: *`. The state-mutating retry and bulk-requeue endpoints are callable from any website via a cross-origin fetch.

The WebSocket server (`new WebSocketServer({ server })`) accepts all incoming upgrade requests unconditionally. No origin check, no token validation, no connection limit. Any client that knows the server address receives the full firehose of job events — leaking job names, queue names, and timing data in a multi-tenant deployment.

## Bulk requeue atomicity and error propagation

The `POST /queues/:name/dlq/requeue` endpoint iterates over job IDs sequentially, performing five Redis operations and one Postgres write per job. If any operation throws mid-loop, the `catch` sends a 500 but the already-processed jobs have been requeued while the remaining ones haven't. The Redis state (status set to 'pending', removed from DLQ list, pushed to work queue) and Postgres state (`markJobPendingForRetry`) are updated in separate calls with no transaction boundary.

A similar issue exists in `POST /jobs/:id/retry`: a failure between the `hset` and the `lpush` leaves a job marked pending in Redis but not enqueued anywhere.

## Incomplete event bus wiring

The `ThroughputChart` component listens for `job.completed` events to count throughput. However, only two event types are ever published:

- `job.created` — from `Producer.enqueue()`
- `job.retried` — from the retry/requeue endpoints

The worker (which handles `job.started`, `job.completed`, `job.failed`, `job.dlq` transitions) has no reference to the event bus. The throughput chart will always show zero, and the live stream will only display creation and retry events. The dashboard's most prominent visualization is non-functional until the worker is wired in.

## EventBus injection via monkey-patch

```typescript
(producer as any).deps.eventBus = eventBus;
```

The `ProducerDeps` interface already declares `eventBus?: NexusEventBus`, so this property could be passed at construction time. The cast to `any` silences the type system and creates a temporal coupling — if code reordering moves `buildRouter` above this line, the producer won't have the event bus and will silently skip event emission.

## DLQ endpoint unbounded response

`GET /queues/:name/dlq` calls `redis.lrange(dlq, 0, -1)` and then fetches the full hash for every returned ID with no pagination. A DLQ with thousands of entries produces an unbounded response and O(n) Redis roundtrips. The queue-jobs endpoint correctly implements limit/offset; the DLQ endpoint should do the same.

## Test coverage

14 new tests cover the six REST endpoints with mock Redis and Postgres. The WebSocket event bus has zero test coverage — no tests verify that publishing an event reaches connected clients. The error handler ordering bug is not caught because the test harness creates its own Express app without the error handler, so tests pass even though production would behave differently.

Not tested: WebSocket broadcast, reconnection logic, event bus lifecycle (close/cleanup), bulk requeue partial failure, DLQ endpoint with large lists.
