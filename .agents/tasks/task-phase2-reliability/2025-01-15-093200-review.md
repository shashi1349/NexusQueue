# At-least-once delivery, retry with backoff, DLQ, and idempotency

Phase 2 replaces the Phase 1 at-most-once BRPOP with BLMOVE into per-worker processing lists, adds explicit ACK on success, exponential backoff retry with a configurable max (default 3), dead-letter queues for exhausted jobs, and idempotency keys on the producer to deduplicate within a 24-hour TTL window. The worker is the heaviest rewrite; the producer gains a check-then-set guard around the idempotency key and the schema gets a nullable column with a partial unique index.

Watch for: Idempotency check-then-set is not atomic — concurrent enqueues with the same key can both pass the guard and create duplicates (confirmed). Retry re-enqueue lives in a `setTimeout` callback that is lost on worker crash or restart (confirmed). The idempotency key is set in a separate Redis command after the MULTI that creates the job, leaving a crash window where the job exists but the guard does not (confirmed).

## High-level view

The worker now atomically moves jobs from the queue list to a per-worker processing list via BLMOVE (`redis.call('BLMOVE', ...)`), giving crash-recovery semantics: a future janitor can scan stale processing lists and re-enqueue orphans. On success the job is removed from the processing list (ACK); on failure the worker decides between retry (backoff + re-push) and DLQ based on the attempts-vs-maxAttempts comparison.

Retry backoff is implemented entirely in-process via `setTimeout`. This means the retry schedule is ephemeral — if the worker process dies while a delayed re-push is pending, that job disappears from both the queue and the processing list, violating at-least-once. A durable alternative (sorted set with score = next-eligible-time) would survive restarts.

Idempotency in the producer uses a non-atomic read-then-write pattern: GET the idempotency key, and if absent, proceed with the full enqueue then SET the key with a 24h TTL. Two concurrent requests with the same key can both observe the key as absent and both create jobs. Additionally, the SET happens after the MULTI that persists the job to Redis and the INSERT to Postgres, so a crash between those steps leaves a job without its dedup guard.

The Postgres schema addition (nullable `idempotency_key` with a partial unique index) provides a database-level backstop against duplicates, which partially mitigates the Redis race — but the resulting constraint violation propagates as an unhandled error rather than returning the existing jobId.

1. **Idempotency race (producer)** — GET-then-SET is not atomic. Use `SET NX` as the gate (set the idempotency key first with NX, and only proceed if that succeeds) to prevent duplicate job creation under concurrency. (confirmed)

Consider a Redis sorted set (ZADD with score = re-eligible timestamp) polled by the loop, or leave the job on the processing list for a janitor to reclaim. (confirmed)
4. **Postgres unique-constraint error unhandled** — The partial unique index on `idempotency_key` will throw on duplicate INSERT if the Redis race is hit, but the producer doesn't catch this specific error to return the existing jobId gracefully. (confirmed)
5. **Failure-path LREM ordering creates loss window** — `handleFailure` removes the job from the processing list *before* the retry re-push or DLQ push. A crash between LREM and the subsequent write loses the job entirely — it's on no list and no janitor can find it. (confirmed)

## Idempotency: race and crash window in the producer

The producer's idempotency logic reads the key, decides it's absent, then creates the job across Postgres and Redis before finally writing the idempotency guard:

```typescript
// Check
const existing = await this.deps.redis.get(redisKeys.idempotency(options.idempotencyKey));
if (existing) return existing;

// ... uuid, INSERT, MULTI ...

// Guard (separate command, after MULTI)
await this.deps.redis.set(redisKeys.idempotency(options.idempotencyKey), id, 'EX', 86400);
```

Two problems emerge. First, the check-and-set is not atomic — under concurrent load, two requests with the same idempotency key can both pass the GET and both proceed through the full enqueue path. The Postgres partial unique index catches the duplicate INSERT at the database layer, but the resulting constraint violation propagates as an unhandled 500 to the caller rather than a graceful "here's your existing jobId" response.

Second, the SET is a standalone command after the MULTI that creates the job. If the process dies between `multi.exec()` and the `redis.set`, the job exists in both Postgres and Redis but the idempotency guard was never written. A subsequent retry by the client will create a second job.

A tighter implementation would use `SET ... NX` as the initial gate: attempt to SET the idempotency key with NX and EX, and only proceed with job creation if the SET succeeded. On NX failure, GET the stored jobId and return it. This eliminates both the race and the crash window in one operation.

## Retry backoff via in-process setTimeout

When a job fails but has retries remaining, the worker updates Postgres and Redis to "pending" immediately, then schedules the re-push to the queue list via `setTimeout`:

```typescript
setTimeout(() => {
  this.deps.redis.lpush(redisKeys.queue(queueName), jobId).catch(() => {});
}, delay);
```

The `.catch(() => {})` swallows LPUSH failures silently — if Redis is temporarily unreachable when the timer fires, the job is lost with no trace. More critically, the setTimeout callback is purely in-process. If the worker restarts (deploy, crash, OOM kill) while a delayed re-push is pending, that job has already been removed from the processing list (the LREM happens before the setTimeout is scheduled) and was never re-added to the queue. It's in "pending" status in both Postgres and Redis but is invisible to any worker. This is effectively a silent job loss, which contradicts the at-least-once guarantee.

A durable alternative: keep the job on the processing list until the re-push actually succeeds, or use a Redis sorted set where the score is the wall-clock time at which the job becomes eligible for retry. The main poll loop (or a dedicated sweep) can ZPOPMIN entries whose score is ≤ now.

## Processing list cleanup ordering asymmetry

On success, the ordering is safe: markJobCompleted(pg) → hset(redis, "completed") → LREM from processing list. If the worker crashes before LREM, a janitor would re-enqueue an already-completed job. The janitor could check status before re-enqueuing, or at-least-once semantics simply accept the double-execution.

On failure, the ordering is inverted: LREM from processing list happens *first*, then the retry/DLQ writes. If the worker crashes after LREM but before `lpush` (retry) or `lpush` to DLQ, the job is gone from every list. The Postgres row is stuck in "active" status from the earlier `markJobActive` call — a zombie that no component will ever pick up. This is the same fundamental gap as the setTimeout concern: the job leaves the safety of the processing list before a durable next-step is committed.

Fix: defer LREM until after the retry re-push or DLQ push succeeds, mirroring the success path's ordering.
