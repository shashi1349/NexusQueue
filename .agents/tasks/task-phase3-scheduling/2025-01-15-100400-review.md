# Advanced scheduling: v2 review after fix pass

The fix commit addresses the five actionable issues from v1. The concurrent scheduler double-promote and double-fire bugs are resolved via standalone ZREM-then-check guards. Cron-fired jobs now INSERT into Postgres before enqueueing in Redis. The `handleFailure` LREM ordering is corrected in both DLQ and retry branches. Rate-limit re-push uses MULTI for atomicity.

Watch for: The promote path's post-ZREM steps (LPUSH + HSET) are no longer wrapped in a MULTI — a crash between ZREM and LPUSH loses the job from the delayed set without placing it in the queue (likely). The `handleFailure` retry branch does Postgres write and HSET before ZADD, so a crash after `markJobDelayed` but before ZADD leaves the job marked delayed in both stores but not actually scheduled for retry (likely).

## High-level view

The ZREM-before-proceed pattern eliminates the double-promote and double-fire races. Only the ZREM winner proceeds; losers skip via `continue`. The concurrency fix is correct.

The `handleFailure` fix moves LREM to after all durable writes in both branches. A crash now leaves the job on the processing list (janitor-recoverable) rather than lost. However, the retry branch's internal ordering (Postgres → HSET → ZADD → LREM) introduces a new narrow crash window between HSET and ZADD where the job is marked delayed but not actually in the delayed sorted set.

The promote path traded the original MULTI's crash atomicity for the ZREM-check concurrency fix, but didn't re-wrap the remaining LPUSH + HSET in a pipeline. The gap is narrow (one await) but real.

## Promote path: ZREM succeeds but LPUSH may not

The v1 fix replaced the MULTI(ZREM + LPUSH + HSET) with a standalone ZREM check followed by separate LPUSH and HSET calls:

```typescript
const removed = await this.deps.redis.zrem(delayedKey, jobId);
if (removed === 0) continue;

// ... priority lookup ...
await this.deps.redis.lpush(targetList, jobId);
await this.deps.redis.hset(redisKeys.job(jobId), { status: 'pending' });
```

The original MULTI provided crash atomicity for the post-ZREM writes. Now, if the process dies between ZREM and LPUSH, the job is removed from the delayed set and never enqueued. The job hash still says `status: 'delayed'`, and since it's gone from the sorted set the scheduler won't re-discover it. No janitor exists today that detects this state (hash says delayed, sorted set doesn't contain it).

The fix is to re-wrap the post-ZREM writes:

```typescript
const removed = await this.deps.redis.zrem(delayedKey, jobId);
if (removed === 0) continue;
// ... priority lookup ...
const multi = this.deps.redis.multi();
multi.lpush(targetList, jobId);
multi.hset(redisKeys.job(jobId), { status: 'pending' });
await multi.exec();
```

The remaining ZREM-to-MULTI gap is one await — unavoidable without Lua, and acceptable for a non-financial workload.

## handleFailure retry ordering

The retry branch now executes:

```typescript
await markJobDelayed(this.deps.pg, jobId);           // 1. Postgres
await this.deps.redis.hset(redisKeys.job(jobId), {   // 2. Redis hash status
  status: 'delayed', ...
});
await this.deps.redis.zadd(delayedKey, score, jobId); // 3. Redis sorted set
await this.deps.redis.lrem(processingKey, 1, jobId);  // 4. Remove from processing
```

A crash between steps 2 and 3 leaves the job marked delayed but absent from the delayed sorted set. The scheduler won't find it (not in sorted set), and the janitor safety net depends on step 4 not having executed — which holds in the crash case. So the janitor can re-enqueue, but only if it exists and handles a job whose hash says 'delayed' while it sits in a processing list. This is an unusual state that a naive janitor (one that just re-LPUSHes stale processing entries) would handle correctly by accident, but a state-aware janitor might skip.
