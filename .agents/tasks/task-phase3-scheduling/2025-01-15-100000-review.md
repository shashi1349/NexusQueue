# Advanced scheduling: delayed jobs, priority queues, cron, and rate limiting

Phase 3 replaces the in-process `setTimeout` retry with a durable Redis sorted set, adds delayed-job enqueue via ZADD, introduces priority queues with weighted fair pulling (6:3:1 ratio across high/normal/low), a Scheduler polling loop that promotes delayed jobs and fires cron entries, a CronManager for recurring job definitions, and a token-bucket rate limiter with both Lua and non-Lua paths. The retry refactor alone is a significant durability improvement over Phase 2's ephemeral setTimeout approach.

Watch for: The Scheduler's promote-and-fire loop reads due jobs then operates on them one-at-a-time without guarding against concurrent Scheduler instances — two schedulers will double-enqueue cron jobs and double-promote delayed jobs (confirmed). The rate-limited job re-push removes the job from the processing list before sleeping, creating a window where neither the worker nor any janitor owns the job (confirmed). The `handleFailure` LREM-before-ZADD ordering inherited from Phase 2 still loses jobs on crash (confirmed).

## High-level view

The delayed-job path splits enqueue into two branches in the Producer: jobs with a `delay` option go through ZADD to a per-queue sorted set with score = wall-clock due-time, while immediate jobs route to priority-specific lists. The Scheduler polls every second and atomically promotes due entries via MULTI(ZREM + LPUSH + HSET). This is a solid replacement for setTimeout-based retry — the job now survives worker restarts because the sorted set is durable.

Priority routing uses separate Redis lists per priority level and a deterministic 10-step rotation in the Worker's pull loop to approximate the 6:3:1 weighting. The fallback to BLMOVE on the normal queue means the worker only blocks when all three priority lists are empty. The non-blocking `rpoplpush` for priority pulls is a pragmatic choice given ioredis doesn't expose multi-list blocking moves, though it means the worker hot-spins when only high-priority jobs exist and the rotation points elsewhere.

The CronManager stores definitions as Redis hashes and schedules next-occurrence scores via cron-parser. The Scheduler fires due cron entries by inlining its own enqueue logic (avoiding a circular dependency on Producer) and rescheduling with the computed next timestamp. This inline enqueue skips Postgres entirely — cron-fired jobs exist only in Redis until a worker picks them up and marks them active.

The rate limiter implements a token-bucket with a Lua script for atomic production use and a non-Lua fallback (HGETALL + HSET) for environments lacking eval. The worker exclusively uses the fallback path today, and the non-atomic fallback has a read-then-write race under concurrent workers sharing the same queue.

<summary>Issues (7)</summary>

1. **Concurrent scheduler double-fire** — No distributed lock or ZREM-before-read guards the cron fire loop. Two Scheduler instances will both see the same due entry and both enqueue jobs. Use ZPOPMIN or ZREM-then-check-result (only proceed if ZREM returned 1) to make cron firing idempotent. (confirmed)
2. **Concurrent scheduler double-promote** — Same pattern for delayed job promotion: ZRANGEBYSCORE returns due jobs, then a separate MULTI does ZREM+LPUSH. A second scheduler that read the same ZRANGEBYSCORE result will also ZREM (getting 0) and LPUSH, creating a duplicate in the queue. Check the ZREM result inside the MULTI or use ZPOPMIN. (confirmed)
3. **Cron-fired jobs skip Postgres** — The Scheduler's `fireCronJobs` writes to Redis only (MULTI with SADD+HSET+LPUSH). No INSERT into the jobs table. If the system relies on Postgres as the durable audit record, cron-fired jobs have no audit trail and `GET /jobs/:id` returns 404. (confirmed)
4. **Rate-limit re-push ordering** — When rate-limited, the worker does LPUSH (back to queue) then LREM (from processing list). If the worker crashes between LPUSH and LREM, the job exists in both the queue and the processing list. A janitor would re-enqueue it again, creating a triple-execution. Reorder to LREM first, or use a MULTI. (confirmed)
5. **handleFailure LREM-before-ZADD crash window** — The failure path removes the job from the processing list before ZADD to the delayed set. A crash between these two operations loses the job permanently. Defer LREM until after ZADD succeeds, or combine them in a MULTI. (confirmed)
6. **Non-atomic rate-limiter fallback under concurrency** — `checkLimitFallback` does HGETALL then HSET as two separate commands. Two workers can both read the same token count, both decide to allow, and both consume "the same" token — exceeding the configured rate. (confirmed)
7. **Weighted pull hot-spin on low-priority-only rotation slots** — When only high-priority jobs exist and the rotation selects `['normal', 'high', 'low']` or `['low', 'high', 'normal']`, the worker does 1-2 failed `rpoplpush` calls before finding the high-priority job. This is functionally correct but wastes 1-2 Redis round-trips per pull on those rotation indices. (possible — only matters at high throughput)

## Scheduler concurrency: double-promotion and double-fire

The Scheduler's `promoteDelayedJobs` first queries all due jobs with ZRANGEBYSCORE, then loops through them issuing a MULTI per job:

```typescript
const dueJobs = await this.deps.redis.zrangebyscore(delayedKey, '-inf', String(now));
for (const jobId of dueJobs) {
  const multi = this.deps.redis.multi();
  multi.zrem(delayedKey, jobId);
  multi.lpush(targetList, jobId);
  multi.hset(redisKeys.job(jobId), { status: 'pending' });
  await multi.exec();
}
```

The ZREM inside the MULTI will return 0 if another Scheduler already removed the entry, but the code does not check the ZREM result — it unconditionally LPUSHes regardless. Two schedulers both see `job-X` in ZRANGEBYSCORE, both issue the MULTI, one gets ZREM=1 and one gets ZREM=0, but both LPUSH. The job appears twice in the queue.

The `fireCronJobs` path has the same issue: ZRANGEBYSCORE reads due entries, then each is rescheduled with ZADD. Two schedulers will both fire the same cron entry and both enqueue a job before the score gets updated. The fix is the same: use `ZPOPMIN` (atomically removes and returns) or check the ZREM return value after `multi.exec()` and skip the enqueue if it returned 0.

## Cron jobs: no Postgres audit trail

The Scheduler inlines its own enqueue logic to avoid importing Producer:

```typescript
const multi = this.deps.redis.multi();
multi.sadd(redisKeys.queueRegistry, queueName);
multi.hset(redisKeys.job(id), { ... });
multi.lpush(targetList, id);
await multi.exec();
```

There is no `insertPendingJob` or `insertDelayedJob` call. The Postgres jobs table never learns about cron-fired jobs. This means the GET /jobs/:id endpoint returns 404 for cron-spawned work, the Postgres audit trail has gaps, and any future janitor that reconciles Redis state against Postgres rows won't find these jobs. If Postgres is intended as the system of record, cron-fired jobs need an INSERT.

## handleFailure crash window (inherited, still present)

Phase 2's review flagged that `handleFailure` removes the job from the processing list before the durable next-step write. Phase 3 replaced `setTimeout` with `redis.zadd(redisKeys.delayed(...))`, which is a major improvement — the retry destination is now durable. But the ordering is still:

```typescript
await this.deps.redis.lrem(redisKeys.processing(this.deps.workerId), 1, jobId);
// ... if crash happens here ...
await this.deps.redis.zadd(redisKeys.delayed(queueName), String(dueAt), jobId);
```

A crash between LREM and ZADD still loses the job. The window is narrower than Phase 2's setTimeout gap, but it exists. Moving LREM after ZADD (or combining them in a MULTI pipeline with the delayed set) would close it.

## Rate-limiter: production vs fallback path mismatch

The Worker unconditionally calls `checkLimitFallback` (the non-Lua path):

```typescript
const result = await this.rateLimiter.checkLimitFallback(this.deps.redis, queueName, rateConfig);
```

The Lua `checkLimit` method is never called in production code. The fallback does HGETALL → compute → HSET as two separate Redis commands. Under multiple workers sharing the same queue, two workers can both read the same token count (say, 1 token remaining), both decide they're allowed, both decrement to 0, and both proceed. The rate limit is violated by a factor equal to the number of concurrent workers. The Lua script exists precisely to prevent this race, but it's unused. If ioredis-mock is the only reason for the fallback, consider detecting eval support at startup and choosing the path dynamically.

## Rate-limited job re-push ordering

When the rate limiter denies execution, the worker pushes the job back to the front of its priority queue and then removes it from the processing list:

```typescript
await this.deps.redis.lpush(targetList, jobId);
await this.deps.redis.lrem(redisKeys.processing(this.deps.workerId), 1, jobId);
await new Promise((r) => setTimeout(r, result.retryAfterMs));
```

Between LPUSH and LREM, the job exists in both the queue and the processing list simultaneously. Another worker can pop it from the queue while the original worker hasn't yet removed it from its own processing list. The sleep afterward adds to the window duration. If a janitor scans the processing list during this window, it may re-enqueue a job that's already back in the queue — leading to double execution.

## Test coverage

The test suite covers the main functional paths well: delayed enqueue, scheduler promotion with priority routing, cron add/remove/list/fire, rate limiter token depletion and refill, and the retry-to-delayed-set behavior.

Not tested: multiple Scheduler instances racing (the double-promotion/fire bug), rate limiter under concurrent worker access, the Lua `checkLimit` path (intentionally skipped due to ioredis-mock limitations, but the function is dead code in production too), the BLMOVE fallback in `pullJob` (only rpoplpush path is hit since tests seed jobs directly), and the rate-limit re-push path's ordering with respect to concurrent workers.
