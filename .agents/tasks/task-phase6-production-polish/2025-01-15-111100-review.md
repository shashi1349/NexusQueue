# Production polish: structured logging, Prometheus metrics, auth, and OpenAPI (v2)

This change layers production concerns onto the existing queue system: structured pino logging replaces all console output, a Prometheus `/metrics` endpoint exposes custom counters and default process metrics, API key and JWT auth gates production endpoints, and Swagger UI serves a hand-maintained OpenAPI 3.0 spec. The v2 pass addressed the timing-attack surface with a `crypto.timingSafeEqual` helper, pinned JWT verification to HS256, and updated `.env.example` with all auth/logging variables.

**Watch for:** The `timingSafeEqual` helper leaks key length through its early-return path — when buffers differ in length it compares `bufA` against itself and returns `false`, which is constant-time for the byte comparison but the overall timing still differs from the equal-length path because `Buffer.from()` allocation scales with string length (possible). No test exists for expired JWT rejection (confirmed).

## High-level view

The timing-safe fixes address the core v1 concerns. The `timingSafeEqual` helper compares buffers of equal length via `crypto.timingSafeEqual` and pads the unequal-length case with a self-comparison. The API key middleware iterates all configured keys without short-circuiting, preventing timing leaks about which key position matched. The login handler applies the same helper to both username and password.

JWT verification pins `algorithms: ['HS256']`, `.env.example` documents all new variables, and dead metrics carry a TODO explaining the cross-process constraint. The remaining concerns are minor: a theoretical length oracle in the timing-safe helper and a missing test for expired tokens.

## timingSafeEqual implementation

The loop in `apiKeyMiddleware` (lines 56-60) iterates all valid keys without breaking on match:

```typescript
let matched = false;
for (const validKey of validKeys) {
  if (timingSafeEqual(key, validKey)) {
    matched = true;
  }
}
```

This prevents revealing which key position matched, but does reveal the number of configured keys through total response time (since each iteration performs a `crypto.timingSafeEqual` call). For a queue system this is immaterial — the number of API keys is not a secret.

The length-mismatch branch allocates only `bufA` and compares it against itself. An attacker who controls input length can observe that requests with the correct key length take marginally longer (the `bufB` allocation and real comparison). For 32+ character API keys this is deep in the noise floor of network jitter. If the system is ever exposed to high-precision local timing (e.g., shared-host IPC), hashing both sides with HMAC-SHA256 before comparison would normalize length.

## Test coverage gap: expired JWT

The test suite does not create a structurally valid, correctly-signed JWT whose `exp` claim is in the past and assert 401. This is the most common production auth failure and a one-line test addition:

```typescript
const expired = jwt.sign({ sub: 'admin' }, 'test-secret', { expiresIn: '-1s' });
```
