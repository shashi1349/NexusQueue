# Production polish: structured logging, Prometheus metrics, auth, and OpenAPI

This change layers production concerns onto the existing queue system. Console output is replaced by structured pino logging (with pretty-print in dev, JSON in production). A Prometheus `/metrics` endpoint is added with custom counters and gauges. Job submission (`POST /jobs`) is gated behind API key auth, dashboard endpoints behind JWT, and a `/auth/login` endpoint issues tokens. Swagger UI at `/docs` serves a hand-maintained OpenAPI 3.0 spec covering all endpoints. The README is rewritten with an architecture diagram, configuration table, and comparison to BullMQ.

**Watch for:** Most Prometheus metrics are defined but never incremented — only enqueue and retry counters are wired up, meaning completed/failed/dlq/duration/depth/workers will always read zero (confirmed). API key validation uses JavaScript `===` and `Array.includes`, which are not constant-time — timing side-channel risk on key comparison (confirmed). The `.env.example` file is not updated with the new auth variables (confirmed).

## High-level view

The logging migration replaces all `console.*` calls with a single `createLogger(service)` factory in `shared/`. pino-http is wired into Express with exclusions for `/health` and `/metrics`. The ESM/CJS interop cast for `pino-http` works but is brittle across module resolution changes.

Auth follows a "disabled-by-default" pattern: when `API_KEYS` or `JWT_SECRET` are absent, middleware calls `next()` unconditionally. This means the system is fully open unless operators explicitly configure four env vars — none of which appear in `.env.example`.

The metrics module defines eight custom Prometheus instruments but only two (`jobs_enqueued_total` and `jobs_retried_total`) are actually incremented anywhere. The remaining six are dead declarations that will emit only zeros. The worker process doesn't import the metrics module at all, so job lifecycle counters can't fire from there.

The OpenAPI spec is hand-maintained alongside route definitions. Drift is possible as the API evolves, and there's no generated validation that they stay in sync.

1. **Dead metrics declarations** — `jobsCompletedTotal`, `jobsFailedTotal`, `jobsDlqTotal`, `jobDurationSeconds`, `queueDepth`, and `activeWorkers` are defined but never incremented. Either wire them into the worker lifecycle or remove them to avoid misleading dashboards.

4. **`.env.example` missing auth variables** — `API_KEYS`, `JWT_SECRET`, `DASHBOARD_USER`, `DASHBOARD_PASSWORD`, `LOG_LEVEL`, and `NODE_ENV` are documented in README but absent from `.env.example`, which is the actual file operators copy.
5. **No JWT `algorithm` restriction on verify** — `jwt.verify(token, secret)` without specifying `algorithms: ['HS256']` will accept any algorithm the library supports, including `none` in some library versions. Pin the algorithm.
6. **Metrics endpoint is unauthenticated** — `/metrics` has no auth, which is fine behind a firewall but exposes internal counters (process memory, CPU, event loop lag via default metrics) to anyone who can reach the server.

## Metrics instrumentation gap

The `server/src/metrics.ts` module exports eight Prometheus instruments. Only `jobsEnqueuedTotal` and `jobsRetriedTotal` have helper functions called from `routes.ts`. The other six are exported as raw `Counter`/`Gauge`/`Histogram` instances but nothing ever calls `.inc()`, `.set()`, or `.observe()` on them.

The worker — where job completion, failure, and DLQ transitions actually happen — doesn't import the metrics module. This is architecturally tricky because the worker runs as a separate process, so it can't share the server's Prometheus registry. The design implicitly assumes a push-gateway or per-process `/metrics` endpoint on the worker, neither of which exists. Until the worker can report its own counters (or the server scrapes state from Redis/Postgres on each `/metrics` call), these instruments are dead code that will produce misleading zero-value time series.

## Auth middleware — timing and fail-open semantics

The validation logic uses `validKeys.includes(key)`, which performs sequential `===` comparisons that leak information about key prefixes through response timing. For a queue system behind a VPN this is low-severity, but for internet-facing deployments it's a real vector. The fix is to iterate with `crypto.timingSafeEqual` (padding shorter strings to match length).

The same issue applies to `loginHandler`, which compares `username !== dashUser` and `password !== dashPass` with `!==`. The password comparison is the higher-value target.

The JWT middleware calls `jwt.verify(token, secret)` without passing an `algorithms` option. The `jsonwebtoken` library defaults to accepting whatever algorithm the token header claims. Pinning `algorithms: ['HS256']` is a one-line defense-in-depth measure that prevents future library upgrades from silently weakening verification.

## pino-http interop fragility

The `pino-http` import uses a double-cast (`pinoHttpModule as unknown as typeof pinoHttpModule.default`) to work around ESM/CJS resolution. This is fragile under `"moduleResolution": "bundler"` or if pino-http ships a different export shape in a minor version. A `// @ts-expect-error` with a comment explaining why would be clearer intent than a double `as unknown as` cast.

## Test coverage gaps

The 17 tests cover metrics endpoint content, API key auth (valid/invalid/disabled), JWT auth (valid/invalid/disabled), login flow (success, failure, not-configured, end-to-end with returned token), and Swagger UI availability.

Not tested: expired JWT rejection, malformed JSON in login body, metrics counter increments after actual enqueue (tests check endpoint content but don't verify counters change), and the `pino-pretty` fallback path in `createLogger`.
