# Load testing infrastructure, Docker packaging, and deployment documentation

Phase 7 adds the operational layer around NexusQueue: k6 load test scripts for benchmarking the enqueue path, multi-stage Dockerfiles for all three services (server, worker, dashboard), a production docker-compose manifest, a GitHub Actions CI pipeline, and comprehensive documentation (ARCHITECTURE.md, DEPLOYMENT.md, README benchmark section). No source code was modified; this is entirely new infrastructure and documentation files.

**Watch for:** Dashboard Dockerfile context mismatch between docker-compose.prod.yml and the CI workflow (confirmed). Worker containers lack health checking, creating a blind spot for orchestration-level restarts (confirmed). CI test job provisions real Redis/Postgres services but the test suite uses ioredis-mock, so the services are unused overhead (likely).

## High-level view

The k6 scripts provide three load profiles (burst, sustained, mixed-priority) targeting the enqueue endpoint, though they only exercise the happy path — no error scenarios, no dequeue-side testing, no assertions beyond HTTP 201. The load test parameters (`sleep(0.1)` per iteration, 500 req/s constant rate) cap actual throughput well below the 10,000 jobs/sec benchmark claim in README.

The Docker packaging uses a consistent multi-stage pattern across server and worker. The dashboard Dockerfile assumes its own directory as build context, but `docker-compose.prod.yml` sets the context to the repo root — this will break the dashboard build in compose while the CI workflow (which correctly uses `dashboard/` as context) passes.

The CI workflow provisions Redis and Postgres service containers for the test job, but the test suite uses ioredis-mock and requires no external services. The services add startup time without coverage benefit.

<summary>Issues (6)</summary>

1. **Dashboard build context mismatch** — `docker-compose.prod.yml` sets `context: .` with `dockerfile: dashboard/Dockerfile`, but the Dockerfile uses relative `COPY . .` expecting `dashboard/` as the context (and the CI job correctly uses `dashboard/` as context). The compose build will copy the entire repo into the image, bloating it and potentially failing at the `npm run build` step if it finds the wrong package.json. Change compose to `context: dashboard` + `dockerfile: Dockerfile`, or rewrite the Dockerfile for root context.

2. **Worker has no health check** — The server Dockerfile defines a HEALTHCHECK directive; the worker Dockerfile does not. Docker Compose `restart: unless-stopped` will restart a crashed worker, but cannot detect a hung worker (stuck event loop, blocked Redis connection). Adding a lightweight health mechanism (file-based liveness probe, or an HTTP endpoint) would let orchestrators distinguish "dead" from "stuck."

3. **CI test services unused** — The test job spins up Redis 7 and Postgres 16 as GitHub Actions services and passes `REDIS_URL`/`DATABASE_URL` env vars, but the test suite uses ioredis-mock. Unless the tests detect real Redis at runtime and switch, the services add ~30s of startup time without benefit. Either remove the services or add an integration test suite that uses them.

4. **Enqueue-only load coverage** — All k6 scripts POST to `/jobs` and check for 201. There are no load tests for the dequeue path, error scenarios, rate-limited queues, or WebSocket endpoints. The benchmark numbers in README claim worker throughput figures (~2,000 jobs/sec per worker) that are not validated by any included test.

5. **No USER directive in Dockerfiles** — Both server and worker run as root inside the container (no `USER` instruction after the runtime stage). A compromised handler could write to the filesystem or escalate within the container. Add `RUN adduser -D app && USER app` or equivalent.

6. **Redis has no authentication in compose** — The Redis service in `docker-compose.prod.yml` has no `requirepass` configured. Any container that joins the `internal` network can access Redis without credentials. Low risk in isolated environments, but worth adding a password for shared infrastructure.

## Dashboard Dockerfile context problem

The `docker-compose.prod.yml` declares the dashboard service with:

```yaml
dashboard:
  build:
    context: .
    dockerfile: dashboard/Dockerfile
```

But `dashboard/Dockerfile` does:

```dockerfile
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build
```

With `context: .` (repo root), the `COPY package.json` grabs the *root* package.json (the workspace orchestrator), not `dashboard/package.json`. The subsequent `npm ci` would install workspace dependencies, and `npm run build` would fail or produce unexpected output because it's not running inside the Vite project.

The CI workflow handles this correctly: `docker build -f dashboard/Dockerfile dashboard/` uses `dashboard/` as the build context. The compose file needs the same treatment — either `context: dashboard` + `dockerfile: Dockerfile`, or refactor the Dockerfile to accept root context.

## Worker container observability gap

The server Dockerfile includes:

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1
```

The worker Dockerfile has no equivalent. Workers are long-running processes that can enter a state where the process is alive but non-functional (Redis connection dropped silently, event loop blocked by a runaway handler). Without a health check, Docker/Compose has no way to restart a stuck worker — it only restarts on process exit.

The existing heartbeat mechanism (the worker writes to Redis every 10s) could be repurposed: a small HTTP server on the worker that returns 200 when the last heartbeat was sent within 30s would be lightweight and consistent with the existing architecture.

## k6 throughput gap vs README claims

`enqueue-load.js` includes `sleep(0.1)` between iterations, which caps each VU to ~10 req/s. With 100 VUs that's ~1,000 req/s maximum — an order of magnitude below the 10,000 jobs/sec claim in the README benchmark section. The `scenarios.js` sustained scenario uses `constant-arrival-rate` at 500/s, still well below the claim.

The README acknowledges these are "expected results" rather than measured output, but a reader encountering the k6 scripts would reasonably expect them to reproduce the documented numbers. Either increase the k6 scenario rates to match, or clarify in README that the included scripts are smoke tests and the benchmark figures come from a separate setup.

## Docker networking and security

The compose file uses a two-network topology: `internal` (Redis, Postgres, server, workers) and `public` (server, dashboard), preventing direct external access to data stores. Redis uses `--maxmemory-policy noeviction` — correct for a queue system where silent data loss is worse than an OOM error.

Redis has no `requirepass` set. Within a Docker bridge network this is low-risk since only compose-defined services can reach it, but any process on the same Docker host that joins the `internal` network gets unrestricted access. For shared infrastructure deployments, adding a Redis password and updating the connection URLs is straightforward.

## CI caching concern

The `actions/cache` configuration caches the `node_modules` directory keyed by `package-lock.json` hash. Caching `node_modules` directly (rather than the npm cache at `~/.npm`) can cause issues with native addons compiled for different platforms or Node versions. For this project (no known native deps), it's functional, but switching to npm's cache directory would be more robust if native dependencies are added later.
